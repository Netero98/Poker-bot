# Аудит безопасности Poker-bot

## Критические уязвимости

### 1. Небезопасный механизм автообновления (RCE)

**Файл:** `poker/tools/update_checker.py:22-55`

Приложение скачивает `.exe`-файл с удалённого сервера и запускает его через `subprocess.call(..., shell=True)` без какой-либо проверки цифровой подписи или контрольной суммы.

**Риск:**
Если сервер обновлений будет скомпрометирован или произойдёт MITM-атака, злоумышленник сможет выполнить произвольный код на компьютерах всех пользователей приложения.

---

### 2. Загрузка нейросетевой модели без проверки целостности

**Файлы:**

* `poker/main.py:159-166`
* `poker/tools/mongo_manager.py:79-89`

Архитектура модели (`JSON`) и бинарные веса скачиваются с удалённого сервера `dickreuter.com:7778` и загружаются через `model_from_json()` с последующей записью на диск.

**Проблемы:**

* отсутствует checksum;
* отсутствует цифровая подпись;
* отсутствует контроль целостности.

**Риск:**
Подмена модели на сервере позволяет доставить вредоносные или модифицированные данные всем клиентам.

---

## Высокий риск

### 3. Hardcoded credentials / публичные учётные данные

**Файлы:**

* `poker/config.ini`
* `poker/config_default.ini`

```ini
login = guest
password = guest
db = https://dickreuter.com:7778/
```

Эти учётные данные поставляются вместе с приложением и используются в:

* `mongo_manager.py`
* `game_logger.py`
* `strategy_handler.py`
* `gui/action_and_signals.py`

**Риск:**
Любой пользователь или сторонний сервис может использовать одинаковые публичные креды для доступа к API.

---

### 4. Передача логина и пароля через URL

**Файлы:**

* `poker/tools/strategy_handler.py`
* `poker/tools/game_logger.py`

Учётные данные передаются в формате:

```txt
?login=guest&password=guest
```

**Риск:**

* утечка в proxy/server logs;
* утечка через browser history;
* утечка через monitoring systems и analytics.

---

### 5. Уязвимая версия PyJWT

**Файлы:**

* `requirements_mac.txt:26`
* `requirements_win.txt:26`

Используется:

```txt
PyJWT==1.7.0
```

Известная уязвимость:

* **CVE-2022-29217**

**Риск:**
Потенциальный bypass проверки токенов в определённых сценариях.

---

### 6. Массовая отправка данных на удалённый сервер

Приложение передаёт на внешний сервер:

* игровые логи;
* скриншоты;
* карты пользователя;
* имя компьютера;
* данные игровых сессий.

Сервер:

```txt
https://dickreuter.com:7778
```

**Риск:**
Пользовательские данные централизованно собираются без полноценной прозрачности и granular consent-механизмов.

---

## Средний риск

### 7. Использование eval() на данных с сервера

**Файл:** `poker/tools/game_logger.py:151`

Обнаружено:

```python
eval(i)
```

где `i` приходит с удалённого API.

**Риск:**
При компрометации сервера возможен Remote Code Execution.

**Рекомендация:**
Заменить на:

```python
ast.literal_eval()
```

---

### 8. Неавторизованный endpoint `/take_screenshot`

**Файл:** `poker/restapi_local.py:34-39`

Любой локальный процесс может вызвать endpoint и сделать скриншот экрана пользователя.

**Риск:**
Локальное шпионское ПО или вредоносные процессы могут использовать API для скрытого получения изображений экрана.

---

### 9. Некорректная CORS-конфигурация

**Файл:** `restapi_local.py:26-31`

Обнаружено:

```python
allow_origins=["*"]
allow_credentials=True
```

**Риск:**
Нарушение CORS security model, потенциальное расширение поверхности атак.

---

### 10. Docker container с host network + X11 socket

Контейнер запускается с:

* `network_mode: host`
* проброшенным X11 socket

**Риск:**
Контейнер получает:

* полный сетевой доступ хоста;
* доступ к GUI/X11;
* возможность взаимодействия с окнами и вводом пользователя.

---

# Общий вывод

Признаков вредоносного ПО, скрытого майнера или намеренно встроенного spyware не обнаружено. Проект выглядит как легитимный покерный бот с клиент-серверной архитектурой.

Основные риски связаны не со злым умыслом разработчика, а с небезопасными архитектурными решениями:

* автообновление без проверки подписи;
* загрузка моделей без верификации;
* использование `eval()` на удалённых данных;
* публичные credentials;
* небезопасный локальный API;
* чрезмерно открытая Docker-конфигурация.

С точки зрения offensive security проект имеет несколько потенциальных RCE-векторов и требует серьёзного hardening перед использованием на основной системе или в production-среде.
